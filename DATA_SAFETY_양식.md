# BizConnect — Google Play 데이터 보안(Data Safety) 양식 초안

> Play Console → 앱 콘텐츠 → **데이터 보안(Data safety)** 설문에 그대로 입력.
> 실제 코드 동작 기준으로 작성됨. **개인정보처리방침(/privacy)과 100% 일치해야 함** (불일치 시 거절).
> ⚠️ 제출 전 반드시 "미해결 과제"(맨 아래) 처리 필요 — 특히 **인앱 회원 탈퇴**.

---

## 1단계: 데이터 수집·보안 (개요)

| 질문 | 답 |
|------|-----|
| 사용자 데이터를 수집/공유하나요? | **예** |
| 전송 중 데이터가 모두 암호화되나요? | **예** (HTTPS/TLS, `sm.on1.kr`) |
| 사용자가 데이터 삭제를 요청할 수 있나요? | **예** — ✅ 인앱 계정 삭제 구현됨(설정 → 회원 탈퇴, `DELETE /api/auth/account`) + 웹 삭제 경로 안내 |
| 계정 삭제 URL | `https://sm.on1.kr/privacy` 에 삭제 요청 절차 안내 (또는 별도 삭제 페이지) |

---

## 2단계: 데이터 유형별 신고

표기: 수집=서버(sm.on1.kr)로 전송 / 공유=제3자 전달 / 필수·선택 / 목적

| 데이터 유형 | 수집 | 공유 | 필수/선택 | 목적 | 비고 |
|------------|------|------|-----------|------|------|
| 이름 | 예 | 아니요 | 필수 | 계정 관리 | 회원가입 |
| 이메일 주소 | 예 | 아니요 | 필수 | 계정 관리 | 로그인 ID |
| 전화번호 | 예 | 아니요 | 필수 | 계정 관리·본인인증 | SMS 인증 |
| **문자(SMS/MMS) 내용** | 예 | 아니요 | **선택** | 앱 기능(웹 연동), AI 분석 | 사용자가 동기화 **동의**한 경우만(`isSmsSyncConsented`) |
| **연락처(주소록)** | 예 | 아니요 | **선택** | 앱 기능(웹 발송 수신자 관리) | 사용자가 동기화 버튼 누른 경우만 |
| **통화기록(통화 정보)** | **아니요** | 아니요 | — | 자동 콜백 대상 번호 식별 | ⭐ **기기 내에서만 처리, 서버 미전송** → "수집 안 함" |
| 기기 ID(FCM 토큰) | 예 | 아니요 | 필수 | 푸시 알림 | |
| 앱 활동(문자 발송 이력) | 예 | 아니요 | 선택 | 발송 이력 웹 연동 | 동의 시 |
| IP 주소 | 예 | 아니요 | 필수 | 보안(비정상 접근 감지) | 자동 수집 |
| **결제/구매 내역** | 예 | **예** | 선택 | 크레딧 충전·구독 | 결제는 **NICE Pay**(제3자) 처리 |
| 위치 | 아니요 | 아니요 | — | — | 수집 안 함 |
| 사진/미디어 | 예(선택) | 아니요 | 선택 | 명함·MMS 첨부 이미지 | 사용자가 첨부한 경우만 |

### 핵심 포인트 (심사관이 보는 곳)
- **통화기록 = "수집 안 함"** 으로 신고. (읽긴 하지만 기기 내에서만 쓰고 서버로 안 보냄.) → 권한 명세의 "기기 내 처리" 주장과 일치.
- 문자·연락처 = **선택(Optional)** + 동의 기반. 강제 아님.
- 제3자 공유는 **결제(NICE Pay)뿐**. 나머지는 자사 서버 수집(=공유 아님).

---

## 3단계: 보안 관행 (Security practices)
- ✅ 전송 중 암호화(TLS)
- ✅ 사용자가 데이터 삭제 요청 가능 (계정 삭제)
- ✅ 비밀번호 BCrypt 해시 저장
- 독립 보안 검토: 없음(체크 안 함)

---

## 제출 전 과제
1. ✅ **인앱 회원 탈퇴 UI** — 설정 → "회원 탈퇴"(이메일+비밀번호 확인) → `DELETE /api/auth/account` 호출 후 로컬 인증정보 삭제. **구현 완료.**
2. ✅ **웹 삭제 경로** — 개인정보처리방침(/privacy) 제7조의2에 "앱 내 삭제 + 이메일 요청(pcon@naver.com)" 명시. **완료.**
3. ✅ **서버 측 실제 삭제** — `DELETE /api/auth/account`가 이메일+비밀번호 검증 후 사용자 관련 16개 테이블(문자이력·연락처·결제·구독·예약·신뢰기기 등) + 사용자 행 실제 삭제하도록 구현. **완료(컴파일 검증).**
4. ✅ **개인정보처리방침에 삭제 절차·문의처** 명시 (제7조의2).
5. ⚠️ **방침 ↔ Data Safety 표 최종 대조** — 제출 직전 한 번 더 눈으로 일치 확인.
6. ⏳ **서버 통합 테스트(권장)** — 실제 계정으로 탈퇴 → DB에서 모든 행 삭제되는지 1회 확인 (로컬 빌드만으론 미검증).
